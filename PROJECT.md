# Recon — Sponsor Integrations

> How Recon uses Kimchi, Neo4j, and Tessl. This is a working document, updated as the build progresses.

This file exists so judges (and anyone reading the repo after the night) can see exactly where each sponsor tool is doing work. Each section answers: **what does this tool do, where, and how do you verify it?**

## 🌶️ Kimchi — the LLM brain (build-time AND runtime)

### What it is (in our context)
Kimchi swaps the brain of Claude Code (or any OpenAI-compatible client) from paid Claude to a free open-source model — Kimi K2.5, 262k context window. Per the Kimchi sponsor guide, this lets an agent "cook for hours" without burning paid credits.

### Where we use it

**1. Build-time — Claude Code runs on Kimchi.**
The agent writing this entire project is using Kimchi as its backend. Zero paid LLM credits consumed during the 2-hour build.

How to verify: in Claude Code, type `/status`. Should show `kimi-k2.5`, not `claude-sonnet-*`.

Setup commands used:
```bash
curl -fsSL https://kimchi.dev/install | bash
kimchi setup     # → paste Cast AI Kimchi key → pick Claude Code → global scope
```

**2. Runtime — deployed Next.js app calls Kimchi.**
The deployed app uses the OpenAI SDK pointed at Kimchi's endpoint for two jobs:

- `lib/cluster.ts` — labels subsystems from extracted function groups (one batched call per `/api/analyze`)
- `app/api/ask/route.ts` — the in-app agent. Two-step flow: classify intent + generate Cypher → compose English answer

Configured via env vars:
```
KIMCHI_BASE_URL=<endpoint from sponsor table at event>
KIMCHI_API_KEY=castai_v1_5d85521464aa76802dd48d9edeb722c1e03a6588944117e89237d0ddecf27a6a_e37d6dc7
KIMCHI_MODEL=kimi-k2.5
```

Client init (Node, runtime):
```typescript
import OpenAI from 'openai';
const kimchi = new OpenAI({
  baseURL: process.env.KIMCHI_BASE_URL,
  apiKey: process.env.KIMCHI_API_KEY,
});
const completion = await kimchi.chat.completions.create({
  model: process.env.KIMCHI_MODEL,
  messages: [...],
});
```

### Why it matters here
- Letting the agent build all night doesn't blow the budget. Critical for a 2h agentic build.
- The runtime app makes 2–3 Kimchi calls per `/analyze` and 2 per `/ask`. At Claude prices this would be costly per repo; at Kimchi prices it's free.
- 262k context lets us send all extracted function names to one labelling call instead of paginating.

### What would break the integration
- `/status` showing `claude-sonnet` — Kimchi setup didn't take. Re-run `kimchi setup`.
- 401 from runtime Kimchi calls — env vars wrong on Vercel. Re-add via `vercel env`.

## 🔵 Neo4j — the structural data layer (build-time MCP + runtime driver)

### What it is (in our context)
Neo4j Aura free tier — graph database in the cloud, 50k nodes / 175k relationships free forever. Stores the structural graph of every repo Recon analyzes.

### Where we use it

**1. Build-time — Claude Code talks to Neo4j directly via MCP server.**
During the 2-hour build, Claude Code has the `mcp-neo4j-cypher` MCP server configured. When the agent needs to inspect the graph (e.g. "how many functions are in here?", "what edge types exist?"), it queries Neo4j directly without writing a throwaway script.

Configured in `~/.claude/settings.json` (per Neo4j sponsor guide page 6):
```json
{
  "mcpServers": {
    "neo4j": {
      "command": "uvx",
      "args": ["mcp-neo4j-cypher@latest"],
      "env": {
        "NEO4J_URI": "neo4j+s://xxxxx.databases.neo4j.io",
        "NEO4J_USERNAME": "neo4j",
        "NEO4J_PASSWORD": "xxx"
      }
    }
  }
}
```

How to verify: in Claude Code, ask "how many nodes are in my neo4j database right now?" — it should run a real Cypher query and answer, not say "I can't access your database."

**2. Runtime — deployed app uses `neo4j-driver`.**
The Next.js app writes and reads the graph via the official driver. Same Aura instance, different access path.

- `lib/graph.ts:pushToNeo4j()` — wipes the DB, inserts files/functions/calls/imports/subsystems for a freshly analyzed repo
- `app/api/ask/route.ts` — runs Cypher queries generated by Kimchi to answer questions

Connection pattern:
```typescript
import neo4j from 'neo4j-driver';
const driver = neo4j.driver(
  process.env.NEO4J_URI!,
  neo4j.auth.basic(process.env.NEO4J_USER!, process.env.NEO4J_PASSWORD!)
);
const session = driver.session();
const result = await session.run(cypher, params);
```

### The graph model

```
NODES
  (:Repo {url, name, sha, analyzed_at})
  (:File {path, language, loc})
  (:Function {qname, name, file_path, start_line, end_line, signature})
  (:Subsystem {name, description, color})

EDGES
  (:File)-[:DEFINED_IN]->(:Repo)
  (:Function)-[:DEFINED_IN]->(:File)
  (:Function)-[:CALLS]->(:Function)
  (:File)-[:IMPORTS]->(:File)
  (:Function)-[:BELONGS_TO]->(:Subsystem)
  (:Subsystem)-[:PART_OF]->(:Repo)
```

### Why it matters here
- The Neo4j PDF lists three reasons graphs help AI: "AI stops guessing about data," "cleanest way to give an AI memory," "real free tier." Recon hits the first one directly — the agent answers questions by running a *real query* on a *real graph*, not by hallucinating.
- Questions like "what are the most-called functions" or "which file imports the most others" are one-line Cypher queries on a graph and miserable on a flat table. The graph is the right substrate.
- The `(:Subsystem)` clustering layer means we can answer "show me the auth code" without keyword search — it's a structural relationship.

### What would break the integration
- "Can't connect to database" — Aura free tier pauses after inactivity. Resume it in the console.
- "Forgot the password" — Aura shows the password once on creation. If lost, must recreate the instance.

## 🟣 Tessl — code-correctness via installable Skills

### What it is (in our context)
Tessl provides installable "Skills" — markdown cheat sheets that teach AI tools (Claude Code etc.) how to use specific libraries correctly. Skills load at startup and the AI reads them when relevant. Solves the problem that LLMs are trained on outdated docs.

### Where we use it

**Skills installed during pre-flight:**

```bash
npm i -g @tessl/cli
tessl init    # in the project folder
tessl install github:neo4j-contrib/neo4j-skills
tessl install github:anthropics/skills
```

**`github:neo4j-contrib/neo4j-skills`** — the official Cypher Skill, named in BOTH the Tessl and Neo4j sponsor PDFs. Teaches Claude Code modern Cypher 25 syntax (e.g. `MATCH path = SHORTEST 3 (a)-[*]-(b)` instead of the deprecated `shortestPath((a)-[*..6]-(b))`). Critical because Recon generates a lot of Cypher — both at build time (the agent writes `pushToNeo4j` query patterns) and at runtime (the in-app agent translates English questions to Cypher).

**`github:anthropics/skills`** — Anthropic's own toolkit of best-practice Skills for Claude Code (recommended at the end of the Tessl PDF). General coding patterns that improve the entire build.

**`.tessl/skills/recon-schema/SKILL.md`** — written tonight, in 5 minutes. Encodes Recon's specific graph schema so any Cypher generation is grounded:

```markdown
---
name: recon-schema
description: When writing Cypher queries for the Recon code-analysis app, follow this schema.
---
# Recon graph schema
[full node/edge model]
Rules:
- qname is the unique identifier for Function nodes
- Use modern Cypher 25 syntax
- Always LIMIT results to <= 50
- Read-only: never CREATE/DELETE/MERGE/SET
...
```

The runtime `/api/ask` route includes this Skill's content verbatim in its system prompt to ground the Cypher generation.

### Why it matters here
- Cypher syntax changed materially in recent versions. Without the Skill, the agent writes old-syntax queries that break on Aura. Visible failure mode.
- The custom `recon-schema` Skill means the agent never invents node labels or edge types — it sticks to what we've defined.
- Per the Tessl PDF: ask the same question before and after installing a Skill and you can *feel* the difference. Demo move: run the same Cypher-generation prompt with/without the Skill and show the audience.

### How to verify
- `tessl list` shows 3 Skills installed
- Ask Claude Code: "Write a Cypher query that finds the 3 shortest paths between two functions using modern Cypher 25 syntax." Result should use `SHORTEST 3 (a)-[*]-(b)`, not `shortestPath()`.

### What would break the integration
- Skill installed but AI ignores it — restart Claude Code fully (Ctrl+C twice, then `claude`). Skills load at startup.
- `tessl: command not found` — `npm i -g @tessl/cli`, possibly with `sudo`.

## The combo — all three working together

Per Tip 3 from the Combined Project Ideas PDF: *"During your demo, say out loud: 'This is Claude Code running on Kimchi's free model, querying my Neo4j graph, with a Tessl skill keeping the queries correct.'"*

In Recon, this exact stack is operating in two places simultaneously:

**Build time:**
```
Claude Code (your terminal)
   │ runs on
   ▼
Kimchi (free Kimi K2.5)
   │ has installed
   ▼
Tessl Skills (neo4j-skills + anthropics/skills + recon-schema)
   │ writes correct Cypher
   ▼
Neo4j MCP server
   │ executes against
   ▼
Neo4j Aura
```

**Runtime (deployed app):**
```
User question in chat box
   │ POST /api/ask
   ▼
Next.js API route
   │ system prompt = Tessl recon-schema Skill
   │ + few-shot examples
   ▼
Kimchi (OpenAI-compatible call, model=kimi-k2.5)
   │ returns Cypher + view classification
   ▼
neo4j-driver runs Cypher
   │ on
   ▼
Neo4j Aura
   │ results back through
   ▼
Kimchi (second call: results → English answer)
   │
   ▼
Frontend updates viz canvas (map / chart / subgraph)
```

Every sponsor's tool is load-bearing in both flows. None is bolted on.
